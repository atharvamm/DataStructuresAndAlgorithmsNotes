Meeting Rooms II

Approach

The solution to this question lies in the fact that you need more meeting rooms when meeting rooms are occupied at the same time. You can check this by checking if the start time of the new meeting is before the current meeting ends.

Brute: We can maintain a list of rooms. Everytime we have a new reservation comapre it against the rooms that are already there in the meeting list and then make a call whether we need a new room or not. The idea is if any rooms meeting has already ended add the new reservations end time there and continue.

Non - Brute : An issue with the brute force approach is as the number of rooms increases, we will spend time going through each room. One thing we can do to optimize this, is, that when we add elements we can add them to the correct position in a sorted list so later we can quickly add and element in logn time and find the meeting room where the reservation might be about to end in O1 time since it is always going to be the first element of this sorted list. An alternate to this sorted list is min-heap, by having a min-heap we dont need to worry about maintaining this list.

Main Ideas

  • Interval questions, always look at the end time of current/stored vs start time of new interval.
  • Definitely sorting by start time helps, and then imagine a timeline in your head and how to use it.
  • Also when we have to maintain multiple elements in sorted order a heap helps.
